## The Agent Loop: Observe, Think, Act

Instead of expecting the model to solve everything in one response, ReAct allows the model to break the problem into smaller steps. Each cycle has three parts: **Thought**, **Action**, and **Observation**. First, the model thinks about what it needs to do. Then it performs an action by calling a tool. Finally, it observes the tool's result and decides what to do next. This cycle repeats until the task is finished.

A working agent needs five important components. 
- **conversation history (message buffer)** so it remembers what has already happened. 
- **tool registry**, which is simply a list of tools it can use, such as search, calculator, file reader, or email sender. 
- **stop condition** so it knows when the task is complete. 
- **turn limit** (for example, 20 or 50 steps) to prevent infinite loops if something goes wrong. 
- it needs a way to convert every tool's output into readable text (an **observation**) that the model can understand in the next iteration.


**observation**. The model never directly sees a database query or an API response. Instead, after a tool finishes, the host program converts the output into text and feeds it back to the model.


The agent loop also introduces new challenges. Since tool outputs may come from the internet or other external sources, they should be treated as **untrusted input**. A malicious webpage could contain instructions trying to manipulate the model. Agents can also fail by making one incorrect decision early, causing a chain of bad actions afterward. Long-running agents may take dozens or even hundreds of steps, making debugging difficult without detailed logs and traces.


## ReWOO and Plan-and-Execute: Decoupled Planning


ReWOO and Plan-and-Execute are advanced ways to build AI agents by separating planning from execution. A normal ReAct agent follows a continuous loop: think → use a tool → observe result → think again → use another tool. This is flexible because the agent can change its plan whenever something unexpected happens, but it has a major drawback: every new step carries the entire previous conversation history, reasoning, and tool results. As tasks become longer, the number of tokens increases, making the process expensive and sometimes confusing for the model.


**ReWOO (Reasoning Without Observations)** solves this by separating the agent into three parts: a **Planner**, **Workers**, and a **Solver**. Instead of thinking and acting repeatedly, the planner first creates a complete plan before any tools are used. The workers then execute the planned steps and collect information. Finally, the solver uses the original question, the plan, and the collected information to create the final answer.


The biggest advantage of ReWOO is `lower token usage`. In ReAct, every step includes previous thoughts, actions, and observations, so the conversation becomes larger as the task continues. ReWOO avoids this because the planner creates a compact plan once, workers only perform their assigned tasks, and the solver receives only the necessary information. 


Another advantage is `better error handling`. In ReAct, if a tool fails in the middle of the process, the agent must figure out how to recover while remembering all previous steps. For example, if a web search fails at step 8, the model may become confused about what it was doing. In ReWOO, each worker is independent. If Worker 3 fails, the solver simply receives an error message for that step and can decide whether to continue, ignore it, or provide a partial answer. This makes failures easier to locate and debug.

ReWOO uses a plan `DAG (Directed Acyclic Graph)` instead of a simple list of steps. A DAG describes dependencies between tasks.


A major idea from ReWOO is planner distillation. Since the planner only creates plans and does not need to understand tool results, it can be trained separately. A large powerful model can generate many high-quality plans, and then a smaller model (for example, a 7B parameter model) can learn to create similar plans. This reduces cost because planning does not always require the largest model.

**Plan-and-Execute** is a broader version of ReWOO introduced by the LangChain community. It follows a similar idea:

1. A planner creates a sequence of tasks.
2. An executor performs those tasks.
3. A replanner can modify the plan if something unexpected happens.

The difference is that ReWOO creates a mostly fixed plan, while Plan-and-Execute allows the system to look at results and update the plan.

**Plan-and-Act** extends this idea for very long tasks such as web browsing, computer control, and mobile applications. These tasks may require 30–50+ actions. A normal ReAct agent may lose track of its goal after many steps. Plan-and-Act focuses on training models with examples of good long-term plans, allowing them to handle complex tasks like navigating websites, filling forms, or operating software.


## Reflexion: Verbal Reinforcement Learning

It is a technique that allows AI agents to **learn from their mistakes without changing their model weights**. Traditional reinforcement learning (RL) improves an AI system by running thousands or millions of attempts, calculating errors, and updating the model using gradients. This requires large datasets, expensive GPUs, and long training cycles. Reflexion takes a simpler approach: when an agent fails, it analyzes the mistake in natural language, stores the lesson, and uses that lesson in the next attempt.


Reflexion gives an AI agent the same ability. Instead of changing the model itself, it changes the context given to the model by adding previous lessons learned from failures.


#### A Reflexion system has three main components:

1. **Actor**: 
The Actor is the main agent that tries to complete the task. It behaves like a normal AI agent using tools, reasoning, and actions.

2. **Evaluator**
The Evaluator checks whether the attempt was successful or not. It provides feedback about the result. There are three common types of evaluators:

    - **Scalar evaluator**: Gives a clear score or result, such as pass/fail. For example, a programming task either passes all tests or fails.
    - **Heuristic evaluator**: Uses predefined rules to detect problems. For example, "the agent repeated the same action five times" or "the task took too many steps."
    - **Self-evaluator**: The AI model judges its own performance. For example, it reviews its answer and decides whether it is correct. This is useful when there is no clear answer, but it is less reliable.


3. **Self-Reflector**: 
The Self-Reflector creates a short explanation of why the attempt failed. It does not simply say "try harder." It creates a useful lesson that can improve the next attempt. This reflection is stored in **episodic memory**.

**Episodic memory** is a storage area containing previous lessons learned from past failures. Before starting the next attempt, the agent reads these memories and uses them as guidance. The model itself is unchanged, but its behavior improves because it has additional context.

A small correction in memory can prevent repeating the same mistake.


Reflexion does not solve every problem. It is less useful when:

- The first attempt already succeeds.
- The failure comes from external issues, such as a broken server or network outage.
- The stored reflections become outdated or incorrect.


A common problem is memory rot. Over time, the agent may collect too many reflections. Some lessons may no longer apply, and the context becomes crowded. To solve this, systems use techniques like:

- Removing old reflections using a time limit (TTL).
- Combining similar reflections.
- Ranking important memories.
- Running a separate cleanup process.


## Tree of Thoughts and LATS: Deliberate Search

Normal language models usually solve problems in a **straight line**. They generate one idea, continue from that idea, and finally produce an answer. This is called a chain-of-thought (CoT) style of reasoning. The problem is that if the model makes a mistake early, everything after that mistake is built on the wrong foundation. The model usually cannot go backward and try another approach.


**Tree of Thoughts (ToT)** and **LATS (Language Agent Tree Search)** solve this problem by giving the model the ability to explore multiple possible solutions, compare them, and choose the best path. Instead of thinking like a single road, reasoning becomes like a tree with many possible branches.

<br/>

The main idea is:

**Generate → Evaluate → Keep good ideas → Discard bad ideas → Continue**

Instead of one reasoning path:

**Thought 1 → Thought 2 → Thought 3 → Answer**

### Tree of Thoughts (ToT)

Tree of Thoughts has three major components:

1. **Generate thoughts**: The model creates several possible next steps.

2. **Evaluate thoughts**: The model then judges each possibility.

    The evaluation can use different methods:   
    - `Method 1: Classification` (Good,Maybe,Impossible)
    
    - `Method 2: Numeric scoring` (Approach A: 9/10,Approach B: 4/10,Approach C: 7/10)

    - `Method 3: Voting`: The model generates several possible solutions and votes on which one looks strongest.



### LATS: Combining ToT, ReAct, and Reflexion
`Tree of Thoughts`: Explore different possibilities.  
`ReAct`: Use tools and observe results.  
`Reflexion`: Learn from failures using written feedback.  

Instead of simply thinking:

```
Think → Act → Observe → Finish
```

it does:

```text
Create possible actions
        |
        ↓
Search possible futures
        |
        ↓
Evaluate which future is best
        |
        ↓
Choose the best path
```

### LATS Roles

LATS gives the language model three jobs.

1. **Policy (Action Generator)**: The policy creates possible next actions.

2. **Value Function (Evaluator)**: The value function estimates:`"How good is this path?"`

3. **Self-Reflector**: If a path fails, the agent writes a lesson

### Monte Carlo Tree Search (MCTS)

LATS uses a technique from game-playing AI called Monte Carlo Tree Search.

It has four steps:

1. **Select**: Choose the most promising existing path using **UCT (upper confidence bound for trees).**
2. **Expand**: Create new possibilities from that point.
3. **Simulate**: Try following one possible path.
4. **Backpropagate**: Send the result back up the tree.


**UCT (upper confidence bound for trees).**: Its purpose is balancing:
- `Exploitation`: "Use what already works."
- `Exploration`: "Try something new." (This branch has not been tested.Maybe it is better.)


**The Cost of Search**: The biggest disadvantage of ToT and LATS is cost. Search can require 100 to 1000 times more computation than normal reasoning.


#### When To Use Tree Search
1. There are many possible solutions
2. Mistakes are expensive
3. There is a reliable evaluator


## Self-Refine and CRITIC — Explained Simply

**Self-Refine** — the AI critiques and improves its own work.

**CRITIC** — the AI uses external tools to verify its work before improving it.


### Self-Refine: One Model Playing Three Roles

The main idea:
`Use the same language model as a writer, critic, and editor.`

The model performs three jobs:

1. **Generate**
2. **Feedback**: It examines its own answer and asks qustions like Is it correct?, Is anything missing?
3. **Refine**


#### The AI is checking itself
This sounds useful, but models can be bad at detecting their own mistakes.language models judge based on patterns. A false statement can sound very convincing. This is called a **self-verification problem**.

### Evaluator-Optimizer (Anthropic, 2024)
Anthropic's Dec 2024 post names this as one of the five workflow patterns. Two roles:

- Evaluator: scores the output and produces a critique.
- Optimizer: revises the output given the critique.

Loop until the evaluator passes. This is Self-Refine/CRITIC in Anthropic's framing.
